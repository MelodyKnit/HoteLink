## 一、AI 功能增强（核心优先级）

### 1.1 管理端 AI 功能从"空壳"到真正调用 LLM

**当前状态**：以下三个管理端 AI 视图已有接口但**未真正调用 LLM**，仅返回固定模板文案或 fallback 文本：

| 视图 | 当前实现 | 问题 |
|------|---------|------|
| `AdminAIReportSummaryView` | 返回固定模板文案 | 未调用 LLM 生成真实分析 |
| `AdminAIReviewSummaryView` | 返回固定模板文案 | 未调用 LLM 生成真实洞察 |
| `AdminAIReplySuggestionView` | 仅返回 fallback | 未调用 LLM 生成回复建议 |

**改进方案**：

- 为每个场景新建独立 Prompt 模板（如 `prompts/report_summary/`、`prompts/review_summary/`、`prompts/reply_suggestion/`）
- 在 Prompt 中注入真实业务数据（营收数据、订单统计、评价文本等）
- 调用 `AIChatService` 的 LLM 能力生成内容
- 增加缓存机制（同一查询条件 + 时间范围的结果可缓存一定时间）
- 增加 AI 生成标记（让管理员知道这是 AI 产出，需人工复核）

**预计改动**：
- 新增 3 组 Prompt 模板（system.j2 + user.j2）
- `PromptTemplateService` 扩展支持新场景
- 3 个 View 增加 LLM 调用逻辑
- 前端无需改动（已有渲染逻辑）

### 1.2 AI 智能定价助手

**场景**：根据历史入住率、季节因素、周边竞品、节假日等数据，为管理员提供房型定价建议。

**功能设计**：

- **入口**：库存日历页（`InventoryView`）新增「AI 定价建议」按钮
- **输入**：房型基础信息、历史价格、历史入住率、日期（是否节假日/周末）、所在城市
- **AI 输出**：
  - 推荐价格区间
  - 定价理由说明
  - 与历史同期对比分析
  - 高 / 低价风险提示
- **Prompt 设计**：注入房型数据 + 历史库存数据 + 区域同星级平均价

**后端实现**：
- 新增 `prompts/pricing_suggestion/` 模板
- 新增 `AdminAIPricingSuggestionView` 接口
- 查询指定房型近 30 天库存数据 + 入住率，注入 Prompt

**前端实现**：
- `InventoryView.vue` 增加 AI 建议弹窗
- 展示推荐价格 + 分析说明

### 1.3 AI 经营分析报告（深度）

**场景**：管理员选择日期范围后，AI 生成结构化的多维度经营分析报告。

**功能设计**：

- **报告维度**：
  - 营收趋势分析与同比/环比
  - 入住率波动原因推测
  - 各房型收益贡献排名
  - 渠道 / 客户来源分析
  - 差评关键词提取
  - 异常指标自动高亮
- **输出格式**：结构化 Markdown，前端渲染为可视化报告
- **交互**：支持追问（"为什么周三入住率下降？""哪些房型利润最高？"）

**后端实现**：
- 新增 `prompts/business_report/` 模板
- 聚合查询订单、支付、库存数据，计算关键指标后注入 Prompt
- 支持流式输出（SSE），提升长报告的用户体验

### 1.4 AI 评价情感分析与自动标签

**当前状态**：评价管理仅支持列表展示和手动回复，缺少自动化分析能力。

**功能设计**：

- **情感评分**：对每条评价自动分析正面 / 中立 / 负面情感
- **关键词提取**：提取评价中的高频关键词（如"干净"、"隔音差"、"服务好"）
- **智能标签**：自动给评价打上分类标签（服务、卫生、设施、位置、餐饮、价格）
- **差评预警**：自动识别严重差评并推送通知给管理员
- **趋势分析**：按时间维度展示评价情感变化趋势

**后端实现**：
- 新增 `prompts/review_analysis/` 模板
- 评价创建后触发 Celery 异步任务进行情感分析
- `Review` 模型新增字段：`sentiment_score`、`sentiment_label`、`auto_tags`（JSONField）
- 分析结果写入数据库，管理端直接查询展示

**前端实现**：
- `ReviewListView.vue` 增加情感标签颜色徽章
- 增加评价分析概览面板（情感分布饼图 + 标签词云）

### 1.7 AI 异常检测与预警

**场景**：系统自动监测运营数据异常并通过 AI 生成解释和建议。

**功能设计**：

- **检测维度**：
  - 入住率突然大幅下降
  - 退订率异常升高
  - 某房型长期空置
  - 差评集中爆发
  - 营收异常波动
- **预警方式**：
  - 仪表盘异常指标红色高亮 + AI 解释文字
  - 站内通知推送给管理员
- **AI 角色**：不做最终判断，只提供参考分析和建议操作

**后端实现**：
- Celery Beat 定时任务：每日凌晨执行数据异常检测
- 新增 `prompts/anomaly_analysis/` 模板
- 异常入 `SystemNotice` 表
- 新增 `AnomalyDetectionService`

### 1.8 AI 多轮对话记忆与持久化

**当前状态**：LLM 调用仅传 system + 单条 user message，无多轮会话历史。订房助手靠前端 `booking_context` 补偿，但通用客服场景无上下文记忆。

**改进方案**：

- **对话历史模型**：新增 `ChatSession` 和 `ChatMessage` 模型
  ```
  ChatSession: user, scene, created_at, updated_at
  ChatMessage: session, role(user/assistant/system), content, tokens_used, created_at
  ```
- **上下文窗口管理**：保留最近 N 轮对话（或按 token 数截断），作为 messages 数组传入 LLM
- **会话管理 API**：
  - `GET /api/v1/user/ai/sessions` — 历史会话列表
  - `GET /api/v1/user/ai/sessions/{id}/messages` — 会话消息
  - `DELETE /api/v1/user/ai/sessions/{id}` — 删除会话
- **前端已有**：`AIChatView` 已实现聊天历史的本地保存/恢复/删除，增加后端持久化后可跨设备同步

### 1.9 AI Function Calling 集成

**当前状态**：LLM 槽位抽取依靠"只返回 JSON"的提示指令，未使用 OpenAI 原生 Function Calling / Tool Use 能力。

**改进方案**：

- 定义以下 Function：
  - `search_hotels(city, keyword, min_price, max_price, star, sort_by)` — 搜索酒店
  - `get_hotel_detail(hotel_id)` — 获取酒店详情
  - `get_room_availability(room_type_id, check_in, check_out)` — 查询可用房型
  - `create_booking_link(hotel_id, room_type_id, check_in, check_out)` — 生成预订链接
  - `get_user_orders(status)` — 查询用户订单
  - `get_order_detail(order_id)` — 获取订单详情
- **优势**：
  - 结构化参数提取更可靠（不依赖 JSON 格式指令）
  - 模型可自主决定调用时机
  - 支持多步 tool_use 编排
  - 减少提示工程复杂度
- **兼容性**：DeepSeek/OpenAI/智谱均已支持 function_call，需在 `build_ai_client()` 层增加 tools 参数传递

### 1.10 AI 用量监控与配额管理

**当前状态**：无 token 计数、无调用日志持久化、无限流或配额控制。

**改进方案**：

- **调用日志模型**：
  ```
  AICallLog: user, scene, provider, model, input_tokens, output_tokens,
             total_tokens, cost_estimate, latency_ms, status, created_at
  ```
- **配额控制**：
  - 每用户每日 AI 调用次数上限（可配置）
  - 每月总 token 预算限制
  - 超出配额时返回友好提示
- **管理端监控面板**：
  - AI 调用总量趋势图
  - 按场景 / 供应商 / 模型分布
  - Token 消耗统计
  - 平均响应时间
  - 异常调用告警
- **成本估算**：根据各供应商定价自动估算月度 AI 成本

**实现方式**：

### 1.14 推理模型能力利用

**当前状态**：配置了 `reasoning_model` 字段（如 `deepseek-reasoner`、`gpt-4o`），但代码中始终使用 `chat_model`，推理模型从未被调用。

**改进方案**：

- **适用场景**识别：复杂分析报告、异常检测根因推理、定价策略建议等需要深度思考的场景
- **自动切换**：在 `AIChatService` 中增加 `use_reasoning=True` 参数，指定场景自动切换到推理模型
- **成本平衡**：仅在高价值分析场景使用推理模型，日常对话仍用 chat_model

## 五、系统架构与技术增强

### 5.1 API 限流与防刷

- DRF throttle 配置：匿名用户 / 认证用户 / AI 接口分别限流
- 登录接口防暴力破解（失败 N 次锁定）
- AI 接口单独限流（每用户每分钟 / 每日上限）
- 返回标准 429 状态码和重试提示

### 5.2 全文搜索引擎

- 当前酒店搜索基于 SQLite/MySQL `LIKE` 查询
- 集成 Elasticsearch 或 MeiliSearch
- 支持拼音搜索、模糊匹配、搜索权重
- 酒店名、地址、描述、设施等多字段搜索

### 5.3 缓存策略优化

- Redis 缓存热门酒店列表、首页汇总数据
- 酒店详情页缓存（含失效策略）
- 字典数据缓存
- AI 分析结果缓存（避免重复调用）

### 5.4 异步任务增强

**当前 Celery 已配置但使用较少。**

- AI 调用异步化（避免阻塞请求线程）
- 报表生成异步化
- 评价情感分析异步化
- 数据导出异步化
- 邮件 / 短信发送异步化
- 定时任务：
  - 每日营收统计
  - 过期优惠券自动标记
  - 超时未支付订单自动取消
  - AI 每日异常检测

### 5.5 接口文档自动生成

- `drf-spectacular` 已在技术栈中但未见配置落地
- 完善 Schema 声明
- 自动生成 Swagger / ReDoc 文档
- 支持前端 TypeScript 类型自动生成

### 5.6 自动化测试覆盖

- 后端单元测试（models、services、views）
- API 集成测试（关键流程端到端测试）
- 前端组件测试（Vitest）
- E2E 测试（Playwright）
- CI/CD 管道集成测试

### 5.7 日志与监控体系

- 结构化日志（JSON 格式）
- 请求链路追踪（request_id 贯穿）
- 错误告警（Sentry 或自建）
- 性能监控（API 响应时间）
- AI 调用链监控
